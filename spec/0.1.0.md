# ac -- Agentic Commits: Hunk-Based Atomic Commits for AI Agents

## 1. Problem Statement

AI agents produce large diffs that should be split into atomic commits -- one logical change per commit. The agent knows *which hunks belong together* but has no reliable way to execute that plan. Existing approaches fail:

| Approach | Failure Mode |
|----------|-------------|
| `git add -p` | Interactive -- agents cannot drive it |
| Manual `git apply` | Agent must build patches, adjust line numbers, handle re-indexing. Off-by-one errors are rampant |
| Full-file `git add` | Cannot split a single file across multiple commits |
| agentic-commits script | Re-indexes per commit -- agent must predict post-commit diffs, which is error-prone |

**The core difficulty:** After committing hunks [0,1] from a file, the remaining hunks renumber. Selecting non-contiguous hunks requires line-number adjustment. Same file across multiple commits means the diff changes entirely after each commit. All of this is deterministic arithmetic -- exactly the work a tool should do, not an agent.

**Real-world scenario:** An agent writes 5 test functions in a single file. The user wants each test in its own commit. The agent can identify which diff hunks correspond to which test, but cannot reliably stage them individually. Today, agents either commit everything together (losing atomicity) or attempt manual patch construction (burning tokens on error recovery).

## 2. Core Principles

### 2.1 Four Foundational Principles

| # | Principle | Definition |
|---|-----------|------------|
| P1 | AX First | Every decision optimizes for the AI agent as primary user |
| P2 | Batch Parity | One commit or ten commits -- same JSON plan, same interface |
| P3 | Minimal Tokens | Agent writes a plan once; tool handles all mechanical work |
| P4 | Agent Plans, Tool Executes | Agent produces the commit plan (JSON). Tool deterministically executes it. No ambiguity, no prompts. |

### 2.2 AX (Agent Experience)

hc is an AX-first tool. The agent's job is to produce a JSON commit plan mapping hunks to commits. The tool's job is to execute that plan correctly, handling all the difficult mechanics: diff parsing, hunk selection, line-number adjustment, patch construction, and sequential staging.

**Principle: One plan, one call.** The agent writes the plan once based on the current diff. `hc` executes all commits sequentially. No intermediate agent involvement.

**Principle: Original indices.** Hunk indices in the plan refer to the *original* diff (before any commits). The agent only needs to look at the diff once. The tool handles re-indexing internally.

**Principle: Actionable errors.** Every error includes a `hint` field telling the agent exactly what to do next.

**Principle: Zero data loss.** No change is ever lost. The tool validates complete coverage before executing and guarantees every hunk ends up in exactly one commit.

## 3. Architecture

### 3.1 System Overview

```
AI Agent  --[writes]--> commit-plan.json --[stdin/file]--> hc  --[git calls]--> repository
             Looks at diff once.            Executes plan.        Stages & commits.
             Assigns hunks to commits.      Re-indexes hunks.     Working tree untouched.
             Done.                          Builds patches.
```

### 3.2 The 1-Call Architecture

The optimal agent workflow:

```
1. Agent runs: git diff -U0                    -> sees all hunks
2. Agent writes commit plan (JSON)             -> maps hunks to commits
3. Agent runs: hc plan.json (or pipes stdin)   -> all commits created
```

One hc call for N commits. The agent never touches `git add`, `git apply`, or `git commit` directly.

### 3.3 The Skill Layer

hc is useless without the skill that teaches the agent how to use it. The skill provides:
- When to activate (agent has uncommitted changes and needs atomic commits)
- How to read the diff and write a commit plan
- Exact hc command to run
- How to interpret output and handle errors

The skill is part of hc's distribution. See Section 9 for full skill design.

### 3.4 Prevent Errors, Don't Handle Them

hc's commit strategy is built on **pre-validation**, not error recovery. The execution model has two distinct phases:

**Phase 1: Validate Everything (before any git mutation)**
1. Parse plan JSON
2. For untracked files referenced in the plan with `hunks`, run `git add -N <path>` to make them visible in `git diff` (see Section 7.1)
3. Run `git diff -U0` to capture the full diff (unstaged changes only -- see Section 7.10 for staged changes handling)
4. Validate every file in the plan exists in the diff (or is untracked for full-file mode)
5. Validate every hunk index is within range
6. Validate **complete coverage**: every hunk in the diff is assigned to exactly one commit (see Section 3.5)
7. Validate no duplicate hunk assignments (same hunk in multiple commits)
8. Validate no empty file arrays (each commit must include at least one file)
9. Validate commit messages are non-empty strings
10. **Sequential dry-run using a temporary index:** Copy the current index to a temporary file. Set `GIT_INDEX_FILE` for all subsequent git commands in this phase. For each commit in plan order: build its patches via delta accumulation, apply them to the temporary index with `git apply --cached --unidiff-zero` (no `--check` -- the apply both validates AND advances the temp index state so subsequent commits see the correct diff). If any apply fails, validation fails. Discard the temporary index after validation.

If ANY validation fails: exit with actionable error. Revert any `git add -N` operations. No real git state mutated. The agent fixes the plan and retries.

**Phase 2: Execute Deterministically (all validation passed)**
1. For each commit: stage files using the same re-diff + fingerprint-match approach (see Section 3.6), then `git commit -m <message>`
2. No error handling needed -- every patch was pre-validated against the correct sequential index state
3. Report results

This two-phase approach means: **if Phase 1 passes, Phase 2 cannot fail** (barring external interference like disk full or concurrent git operations). The Phase 1 dry-run faithfully simulates Phase 2 by applying patches sequentially to a temporary index, so the line-number adjustments and patch constructions are identical. The agent never sees partial failures or needs recovery logic.

### 3.5 Complete Coverage Validation

Every hunk in the working tree diff must be assigned to exactly one commit. This guarantees zero data loss.

**Validation rules:**
1. For each file in `git diff -U0`: if the file has hunks that support hunk-select mode, every hunk index must appear in exactly one commit
2. A file can appear in the plan without `hunks` (full-file mode) -- this accounts for all its hunks
3. A file can appear across multiple commits with `hunks` -- the union of all hunk indices must equal the full set `[0..N-1]`
4. If a file in the diff is NOT in the plan at all: validation error

**Untracked files exception:** New files that exist in the working tree but are not in the diff AND not in the plan are ignored (they are not part of the current change set). Only files that appear in `git diff` or `git diff --cached` must be accounted for.

**Opt-out: `allow_unplanned` field.** If the agent intentionally wants to leave some changes uncommitted (e.g., work-in-progress on a separate feature), the plan can include:

```json
{
  "allow_unplanned": ["wip_file.go", "experiments/*"],
  "commits": [...]
}
```

Paths in `allow_unplanned` are excluded from coverage validation. Matching rules:
- Exact paths: `"wip.go"` matches only that file
- Glob patterns: uses Go's `filepath.Match` syntax (`*` matches any non-separator sequence, `?` matches one non-separator character). `**` is not supported.
- Patterns are matched against file paths relative to the repo root
- Example: `"experiments/*"` matches `experiments/foo.go` but NOT `experiments/sub/bar.go`
- Non-matching patterns are silently ignored (no warning or error)
Without this field, every change must be planned.

### 3.6 Core Algorithm: Single-Pass Delta Accumulation

This is the critical algorithm, derived from Git's own `add-patch.c` (`reassemble_patch`):

```
Phase 1 (Validation & Patch Construction):
  1. For untracked files with hunks in plan: git add -N <path>
  2. Run: git diff -U0
  3. Parse all files and their hunks (using go-gitdiff)
  4. Index: file_hunks[path] = [H0, H1, H2, ...]
  5. Compute fingerprints for all hunks (using ordered content -- see 3.8)
  6. Validate complete coverage, no duplicates, no empty commits
  7. Sequential dry-run using temporary index (GIT_INDEX_FILE):
     Copy the current index to a temp file. Set GIT_INDEX_FILE for all commands.
     For each commit in the plan (in order):
       For each file in the commit:
         If full-file mode: git add <path> (to temp index)
         If hunk-select mode:
           a. Run: git diff -U0 -M -- <path> (against temp index)
           b. Match original hunks to current diff by content fingerprint
           c. Build adjusted patch from current diff positions
           d. Apply: git apply --cached --unidiff-zero (to temp index)
       NOTE: No explicit "commit simulation" is needed. git apply --cached
       already mutates the temp index. The accumulated patches in the temp
       index correctly represent the post-commit state for subsequent diffs.
       The key insight: `git diff -U0 -- <path>` against the temp index
       already shows only the remaining unstaged hunks.
     If all patches apply cleanly: validation passes
     Discard temporary index (rm temp file, unset GIT_INDEX_FILE)

Phase 2 (Execution -- identical logic, real index):
  For each commit in the plan:
    For each file in the commit:
      If full-file mode: git add <path>
      If hunk-select mode:
        a. Run: git diff -U0 -- <path>  (fresh diff against current index)
        b. Match original hunks to current diff (by content fingerprint)
        c. Build adjusted patch from current diff positions
        d. Pipe patch to: git apply --cached --unidiff-zero
    git commit -m <message>
```

**Why the temporary index in Phase 1?** The dry-run must faithfully simulate Phase 2. After commit N changes the index, the remaining hunks have different line numbers. By applying patches sequentially to a temporary index, Phase 1 validates the *exact same* patches that Phase 2 will produce. This is what makes the guarantee "if Phase 1 passes, Phase 2 cannot fail" hold.

**Why re-diff per commit?** After commit N changes the index, line numbers shift. Re-diffing gives accurate positions for the current state. Content fingerprinting maps the original hunks to their new positions. Both phases use the same re-diff + fingerprint-match logic.

**Why `-U0`?** Zero-context diffs eliminate context-mismatch failures entirely. Each hunk is self-contained. Combined with `--unidiff-zero`, this is the most robust staging approach.

### 3.7 Delta Accumulation

When selecting non-contiguous hunks from a file, the `+` side line numbers must be adjusted for skipped hunks. The algorithm (from Git's `add-patch.c`):

```
delta = 0
for each hunk in file (in order):
  if hunk is SKIPPED:
    delta += old_count - new_count
  if hunk is SELECTED:
    adjusted_new_start = new_start + delta
    emit hunk with adjusted new_start
```

The `old_start` is never adjusted -- it always refers to the original file.

**Worked example:** File has 4 hunks:

| Hunk | old_start | old_count | new_start | new_count | Effect |
|------|-----------|-----------|-----------|-----------|--------|
| H0   | 5         | 2         | 5         | 5         | +3 lines (replaces 2 with 5) |
| H1   | 20        | 3         | 23        | 0         | -3 lines (pure deletion) |
| H2   | 30        | 1         | 27        | 4         | +3 lines (replaces 1 with 4) |
| H3   | 50        | 0         | 53        | 2         | +2 lines (pure addition) |

**Select [0, 2] (skip H1, H3):**

| Step | Hunk | Action  | delta before | Calculation | delta after | Emitted new_start |
|------|------|---------|-------------|-------------|-------------|-------------------|
| 1    | H0   | SELECT  | 0           | --          | 0           | 5 + 0 = **5**     |
| 2    | H1   | SKIP    | 0           | 0 + (3 - 0) = 3 | 3      | --                |
| 3    | H2   | SELECT  | 3           | --          | 3           | 27 + 3 = **30**   |
| 4    | H3   | SKIP    | 3           | 3 + (0 - 2) = 1 | 1      | --                |

Result: H0 emitted at `@@ -5,2 +5,5 @@`, H2 emitted at `@@ -30,1 +30,4 @@`.

### 3.8 Content Fingerprinting for Hunk Matching

After committing some hunks, the remaining diff has different hunk indices and line numbers. The tool maps original hunks to their current positions using content fingerprints.

**Canonical fingerprint algorithm (normative):**

```
fingerprint(hunk) = SHA-256(
  for each deletion line (in original order): "-" + line_content + "\n"
  +
  "||"
  +
  for each addition line (in original order): "+" + line_content + "\n"
)
```

Lines are in their **original order** (not sorted). Each deletion line is prefixed with `-`, each addition line with `+`, and each line is terminated with `\n`. The literal bytes `||` (no surrounding newlines) separate deletions from additions. The result is hex-encoded.

**Note:** Earlier versions of this spec used sorted lines. Sorting is intentionally avoided because reordering lines within a function (a common refactoring) would produce identical fingerprints for genuinely different hunks.

At startup, compute fingerprints for all original hunks. Before each commit (in both Phase 1 dry-run and Phase 2 execution), re-diff and compute fingerprints for current hunks. Match by fingerprint equality.

**Edge case: identical hunks.** If two hunks have identical content (e.g., adding the same import in two places), disambiguate using this algorithm:
1. For each unmatched original hunk, compute `|original.old_start - candidate.old_start|` for each candidate current hunk with the same fingerprint
2. Assign the candidate with the **smallest absolute difference** in `old_start`
3. If tied (two candidates equidistant), select the candidate with the **smaller `old_start`**
4. Matching is **greedy**: process original hunks in order, and once a current hunk is matched, it is removed from the candidate pool
5. If no candidate remains for an original hunk: emit an error with hint "Hunk content changed between validation and execution. Re-run `hc diff` and rebuild the plan."

### 3.9 Why Not Alternative Approaches

| Approach | Why Not |
|----------|---------|
| `git hash-object` + `update-index` | Must compute full file content per commit. More code, same correctness risk, harder to debug |
| `libgit2` / `git2go` | CGo dependency. Breaks cross-compilation. Maintenance burden. |
| Single-pass without re-diff | After commit N, old-side line numbers are stale. Patch application fails. |
| Context diffs (`-U3`) | Adjacent hunks merge. Context from one hunk conflicts with skipped hunk. Zero-context eliminates this class of bugs. |

## 4. Commit Plan Format (Input)

### 4.1 Schema

```json
{
  "commits": [
    {
      "message": "test(auth): add token expiry test",
      "files": [
        {"path": "auth_test.go", "hunks": [0, 1]},
        {"path": "testutil.go"}
      ]
    },
    {
      "message": "test(auth): add token refresh test",
      "files": [
        {"path": "auth_test.go", "hunks": [2, 3]},
        {"path": "handler_test.go"}
      ]
    }
  ]
}
```

With `allow_unplanned`:

```json
{
  "allow_unplanned": ["wip_feature.go", "experiments/*"],
  "commits": [
    {
      "message": "feat(auth): add login endpoint",
      "files": [
        {"path": "auth.go"},
        {"path": "auth_test.go"}
      ]
    }
  ]
}
```

### 4.2 Field Definitions

#### Root

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `commits` | array | Yes | Ordered list of commits to create |
| `allow_unplanned` | string[] | No | File paths/globs excluded from complete coverage validation. Uses Go `filepath.Match` syntax (no `**` support). Each pattern is matched against file paths relative to repo root. |

#### Commit

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `message` | string | Yes | Git commit message. Must be non-empty. Multi-line messages are supported (first line is the subject). Passed to `git commit -m` as-is. |
| `files` | array | Yes | Files to include in this commit. Must contain at least one entry. |

#### File Entry

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `path` | string | Yes | File path relative to repo root |
| `hunks` | int[] | No | Hunk indices (non-negative integers) into the original diff. If omitted or empty array `[]`, stages the entire file (full-file mode). The tool sorts indices internally; ordering in the array does not matter. |

### 4.3 File Staging Strategies (auto-detected)

| JSON | Strategy | When |
|------|----------|------|
| `{"path": "file.go"}` | Full file: `git add file.go` | Entire file in one commit. Works for new, modified, and deleted files. |
| `{"path": "file.go", "hunks": [0, 2]}` | Hunk select: extract `@@` blocks from `git diff -U0`, adjust line numbers, `git apply --cached --unidiff-zero` | Split file across commits |

### 4.4 Design Decisions

**Original indices (Design B).** Hunk indices in the plan refer to the diff state *before any commits*. The agent reads `git diff -U0` once and assigns indices. The tool handles re-indexing after each commit internally. This is critical for AX: the agent does the semantic work (deciding which hunks go together), the tool does the mechanical work (adjusting line numbers).

**Complete coverage by default.** Every change must be accounted for. This prevents the silent data loss that plagued earlier tools. The `allow_unplanned` escape hatch exists for legitimate partial-commit scenarios but requires explicit opt-in.

**No `action` field.** New files, modified files, and deleted files are all handled automatically. `git add` handles new/deleted files in full-file mode. `git diff -U0` produces valid hunks for modifications and deletions. The tool detects the file state from git.

**No `type` field for binary files.** Binary files cannot be split into hunks. If a binary file appears with `hunks`, the tool errors with a hint. Without `hunks`, it uses full-file mode (`git add`).

**Array-only commits.** The plan is always an array of commits, even for a single commit. Batch parity (P2).

|--------|---------|------|

### 5.3 `hc run <plan>`

Execute the commit plan. Creates commits sequentially, handling all hunk re-indexing automatically.

**Preconditions:**
- Must be inside a git repository
- Staging area must be clean (no pre-existing staged changes -- hard error, see Section 7.10)
- Plan JSON must be valid
- Working tree must have unstaged changes or untracked files referenced in the plan (checked after plan parsing, since untracked files are only discovered from the plan)

**Behavior (Phase 1 -- Validation):**

See Section 3.4 for the full 10-step Phase 1 validation. Summary:
1. Environment checks (git repo, clean staging area)
2. Parse plan, validate structure (non-empty commits, messages, files, paths)
3. `git add -N` for untracked files with hunks
4. Capture diff with `git diff -U0 -M`
5. Validate plan against diff (file existence, hunk ranges, complete coverage, no duplicates)
6. Sequential dry-run using temporary index (see Section 3.6)

If validation fails: print error with hint, exit 2. Revert any `git add -N`. No git state changed.

**Behavior (Phase 2 -- Execution):**
1. For each commit: stage files, run `git commit -m <message>`
2. Report result per commit
3. Report summary

**Output (TTY):**


**JSON field reference for each file entry:**

| Field | Type | Present |
|-------|------|---------|
| `path` | string | Always |
| `hunks` | array | Always (may be empty for binary files) |
| `is_new` | bool | Only if true |
| `is_deleted` | bool | Only if true |
| `is_renamed` | bool | Only if true |
| `old_path` | string | Only if renamed |
| `is_binary` | bool | Only if true |
| `is_untracked` | bool | Only if true (untracked file not yet in git) |

**TTY formatting rules:**
- If `old_count == 0`: `(+new_count lines)`
- If `new_count == 0`: `(-old_count lines)`
- Otherwise: `(-old_count +new_count lines)`
- Append `, new file` if `is_new`, `, deleted` if `is_deleted`

This command exists purely for agent convenience (P1). The agent can also read `git diff -U0` directly, but `ac diff` provides pre-parsed, indexed output that costs fewer tokens to process.

**Untracked files:** `hc diff` also lists untracked files (via `git ls-files --others --exclude-standard`) as entries with `is_new: true`, `is_untracked: true`, and an empty hunks array `[]`. These are valid targets for full-file mode in plans. `hc diff` does NOT run `git add -N` for untracked files (that is only done by `hc run`). Note: untracked files cannot be split into hunks (see Section 7.1).

**No changes:** If there are no unstaged changes and no untracked files, `hc diff` exits with code 2 and error `no uncommitted changes`.

**Staged changes warning:** `ac diff` emits a warning (to stderr) if `git diff --cached` shows pre-staged changes, since `ac run` will reject them. This is informational only -- `ac diff` still outputs the diff.

**Note:** `hc diff` shows the same diff that `hc run` will use for validation. If the staging area changes between `hc diff` and `hc run`, hunk indices may not match. Always run `hc diff` immediately before writing the plan.

### 5.6 Exit Codes

| Code | Meaning | When |
|------|---------|------|
| 0 | Success | All commits created (run), or plan valid (dry-run) |
| 2 | Validation error | Invalid plan, incomplete coverage, hunk out of range, patch dry-run failed, not a git repo, no changes, dirty staging area |
| 3 | Execution error | Unexpected git command failure during Phase 2 execution (post-validation) |

**No exit code 1 (partial failure).** hc's pre-validation strategy means execution either fully succeeds (exit 0) or is never attempted (exit 2). Exit 2 covers ALL pre-execution failures including environment checks (not a git repo, no changes). Exit 3 is reserved for unexpected post-validation failures during execution, which indicate external interference, not a plan problem.

## 6. Error Handling

### 6.1 Structured Errors

Every error includes three fields (following tp's pattern):

```json
{
  "error": "hunk index 5 out of range for auth_test.go (has 4 hunks, indices 0-3)",
  "code": 2,
  "hint": "Run 'hc diff --json' to see available hunks and adjust your plan."
}
```

| Field | Type | Description |
|-------|------|-------------|
| `error` | string | What went wrong |
| `code` | int | Exit code |
| `hint` | string | What the agent should do next |

### 6.2 Validation Errors (Exit 2 -- before any git mutation)

| Scenario | Error | Hint |
|----------|-------|------|
| Not a git repo | `not a git repository` | `Run hc from inside a git repository.` |
| No changes | `no uncommitted changes` | `There is nothing to commit.` |
| Pre-staged changes | `staging area is not clean` | `Run 'git reset HEAD' first. hc requires a clean staging area.` |
| Invalid JSON | `plan parse error: <json error>` | `Check JSON syntax. Plan must have a "commits" array.` |
| Missing `commits` field | `plan missing "commits" array` | `Plan must have a "commits" array with at least one commit.` |
| Empty commits array | `plan has no commits` | `Add at least one commit to the "commits" array.` |
| Empty commit message | `commit 0 has empty message` | `Each commit must have a non-empty message string.` |
| Empty files array | `commit 0 has no files` | `Each commit must include at least one file.` |
| File not in diff | `file "foo.go" has no changes in the working tree` | `Only include files with uncommitted changes. Run 'hc diff' to see changed files.` |
| Hunk index out of range | `hunk index 5 out of range for foo.go (has 4 hunks, indices 0-3)` | `Run 'hc diff --json' to see available hunks.` |
| Negative hunk index | `hunk index -1 is invalid for foo.go` | `Hunk indices must be non-negative integers.` |
| Duplicate hunk in same file entry | `hunk index 0 duplicated in foo.go (commit 0)` | `Remove the duplicate index from the hunks array.` |
| Empty file path | `file path is empty in commit 0` | `Each file must have a non-empty relative path.` |
| Absolute file path | `file path "/etc/foo" must be relative to repo root` | `Use a relative path (e.g., "src/foo.go").` |
| Path traversal | `file path "../../etc/passwd" contains ".." traversal` | `Use paths relative to the repo root without ".." components.` |
| Duplicate hunk assignment | `hunk 2 of foo.go assigned to both commit 0 and commit 2` | `Each hunk must appear in exactly one commit.` |
| Incomplete coverage | `foo.go hunks [2, 3] not assigned to any commit` | `Assign foo.go hunks 2,3 to a commit, or add "foo.go" to allow_unplanned.` |
| Unplanned file | `bar.go has changes but is not in the plan` | `Add bar.go to a commit, or add "bar.go" to allow_unplanned.` |
| Binary file with hunks | `foo.png is a binary file and cannot be split into hunks` | `Remove the "hunks" field to stage the entire file.` |
| Mixed staging modes | `foo.go uses full-file mode in commit 0 and hunk-select in commit 1` | `A file in full-file mode stages everything. Use hunk-select mode in all commits, or put the file in exactly one commit with full-file mode.` |
| Full-file in multiple commits | `foo.go appears in full-file mode in commits 0 and 2` | `A file in full-file mode can only appear in one commit. Use hunk-select mode to split across commits.` |
| Patch dry-run failed | `patch validation failed for foo.go hunks [0,1]: <git stderr>` | `This may indicate a diff parsing issue. Run 'hc diff' to inspect the current state.` |

### 6.3 Execution Errors (Exit 3 -- unexpected post-validation failure)

These should never happen if validation passed. They indicate external interference:

| Scenario | Error | Hint | Recovery |
|----------|-------|------|----------|
| `git apply` failure | `git apply failed for foo.go: <git stderr>` | `Working tree may have changed during execution. Run 'git reset HEAD --' and retry.` | hc runs `git reset HEAD --` to unstage all files for this commit |
| `git commit` failure | `git commit failed: <git stderr>` | `Staging is intact. If a pre-commit hook failed, fix the issue and run 'git commit -m "<message>"' manually, then re-plan remaining changes.` | hc does NOT reset staging (files are correctly staged; only the commit itself failed) |

**On execution error:** hc reports which commits succeeded (with SHAs) and which failed, then exits 3. Recovery behavior depends on the failure type:
- **`git apply` failure:** hc runs `git reset HEAD --` (with trailing `--`) to unstage the partially-staged files for the failed commit. Previously created commits are NOT affected.
- **`git commit` failure:** hc leaves the staging area intact. The files are correctly staged; only the commit command failed (e.g., pre-commit hook rejection). The agent can fix the issue and commit manually.

Additionally, hc reverts any remaining `git add -N` (intent-to-add) entries that were not consumed by successful commits, to avoid leaving orphaned index entries.

In both cases, the agent has full information to recover.

```json
{
  "committed": 2,
  "total": 3,
  "error": "git commit failed for commit 2: pre-commit hook rejected",
  "hint": "Fix the pre-commit hook issue. Commits 0-1 are done. Re-plan remaining changes.",
  "commits": [
    {"index": 0, "message": "...", "sha": "abc1234", "status": "committed"},
    {"index": 1, "message": "...", "sha": "def5678", "status": "committed"},
    {"index": 2, "message": "...", "status": "failed", "error": "pre-commit hook rejected"}
  ]
}
```

## 7. Edge Cases

### 7.1 New Files (untracked)

New files appear in `git diff` only if they have been `git add`ed at least once, or with `git diff --no-index`. The tool detects untracked files using `git ls-files --others --exclude-standard -- <path>`. Files excluded by `.gitignore` are not considered untracked and will produce a validation error if referenced in the plan.

For untracked new files referenced in the plan:

- **Full-file mode** (`{"path": "new.go"}`): Uses `git add new.go`. Works for untracked files.
- **Hunk-select mode** (`{"path": "new.go", "hunks": [0]}`): The tool runs `git add -N new.go` (intent-to-add) before Phase 1 diff capture to make hunks visible in `git diff -U0`, then proceeds with normal hunk selection. **Important:** `git add -N` followed by `git diff -U0` produces the entire new file as a **single hunk** (index 0). Splitting a new file across multiple hunks is not possible with `-U0` diffs. If the agent needs a new file in a specific commit, use full-file mode.

### 7.2 Deleted Files

- **Full-file mode** (`{"path": "deleted.go"}`): Uses `git add deleted.go` (stages the deletion).
- **Hunk-select mode**: Splitting a deletion across commits is technically possible (delete some functions in commit 1, rest in commit 2) and uses the same hunk-select algorithm. All hunks in a deleted file are pure deletions (`-` lines only).

### 7.3 Renamed/Moved Files

hc runs `git diff -U0 -M` (with rename detection enabled, using git's default 50% similarity threshold). Rename detection may vary by git version and config (`diff.renames`, `diff.renameLimit`). The plan should use the **new path**:

```json
{"path": "new/location.go", "hunks": [0, 1]}
```

**Coverage validation:** When git detects a rename, the file appears as a **single entry** under the new path in the diff. The old path is NOT a separate entry for coverage validation. The plan only needs the new path.

If the agent uses the old path for a renamed file, the tool emits a validation error: `file "old/location.go" was renamed to "new/location.go"` with hint `Use the new path "new/location.go" in your plan.`

The tool includes rename headers (`rename from`, `rename to`) in the generated patch for the **first commit in plan order** that references the renamed file (parallel to mode change handling in Section 7.4). Subsequent commits for the same file omit rename headers. Full-file mode uses `git add new/location.go && git add old/location.go` (which stages both the deletion of the old path and creation of the new).

### 7.4 Permission (Mode) Changes

Mode changes (`100644` -> `100755`) appear as separate diff metadata. If a file has both mode changes and content changes:
- The mode change header (`old mode`/`new mode`) is included in the patch for the **first commit in plan order** that references the file
- Subsequent commits for the same file omit the mode change header (the tool tracks which files have already had their mode change emitted)
- This is automatic behavior of patch construction, not a validation rule the plan must satisfy
- In full-file mode (`git add`), the mode change is always included automatically by git

### 7.5 Binary Files

- **Full-file mode**: Works normally via `git add`
- **Hunk-select mode**: Validation error with hint (see Section 6.2)

### 7.6 File in Multiple Commits

The primary use case. Example: `auth_test.go` has 4 hunks, split across 2 commits.

- Complete coverage requires hunks [0,1,2,3] to be fully assigned across commits
- Commit 1 takes hunks [0,1]. Tool re-diffs after commit, matches remaining hunks [2,3] by content fingerprint for commit 2
- The agent writes indices based on the *original* diff. The tool remaps internally.

### 7.7 Adjacent Hunks in `-U0` Mode

With zero context, hunks are never adjacent (no context lines to overlap). Each hunk specifies exact line ranges. This eliminates an entire class of bugs present in context-based approaches.

### 7.8 No-Newline-at-End-of-File

The `\ No newline at end of file` marker in diffs is metadata, not a content line. The tool preserves it when constructing patches. It does not count toward `@@` line counts.

### 7.9 Empty Hunks

A hunk with `@@ -5,0 +5,0 @@` (zero old lines, zero new lines) is filtered out during parsing. It represents no change.

### 7.10 Staged Changes

hc requires a **clean staging area** (no pre-existing staged changes). This is a hard precondition, not a warning:

- hc checks `git diff --cached` before Phase 1. If it produces output, hc exits with error (exit 2): `staging area is not clean` with hint `Run 'git reset HEAD' first. hc requires a clean staging area.`
- Rationale: `git diff -U0` (used for hunk indexing) only shows unstaged changes. Pre-staged changes would be invisible to the plan but silently included in the first `git commit`, producing incorrect results. Making this a hard error eliminates an entire class of bugs.
- The skill instructs agents to always start with a clean staging area

### 8.4 Delta Accumulation Algorithm (Go)

This is the most critical function. Derived from Git's `add-patch.c`:

```go
// patch/build.go
// BuildPatch constructs a git patch for selected hunks.
// In Phase 1 dry-run: allHunks = original diff hunks, selectedIndices = original indices.
// In Phase 2: allHunks = CURRENT diff hunks (from re-diff), selectedIndices = current
// indices obtained by fingerprint-matching original hunk indices to current positions.
func BuildPatch(file FileDiff, selectedIndices []int, allHunks []Hunk) ([]byte, error) {
    selected := make(map[int]bool, len(selectedIndices))
    for _, idx := range selectedIndices {
        selected[idx] = true
    }

    var buf bytes.Buffer

    // Write file headers (handle new, deleted, renamed files)
    aPath := file.Path
    if file.IsRenamed {
        aPath = file.OldPath
    }
    fmt.Fprintf(&buf, "diff --git a/%s b/%s\n", aPath, file.Path)
    if file.IsNew {
        fmt.Fprintf(&buf, "new file mode %s\n", file.NewMode) // e.g., "100644" or "100755"
    } else if file.IsDeleted {
        fmt.Fprintf(&buf, "deleted file mode %s\n", file.OldMode)
    } else if file.OldMode != "" && file.NewMode != "" && file.OldMode != file.NewMode {
        // Mode change only for existing files (not new/deleted)
        fmt.Fprintf(&buf, "old mode %s\nnew mode %s\n", file.OldMode, file.NewMode)
    }
    if file.IsRenamed {
        fmt.Fprintf(&buf, "rename from %s\nrename to %s\n", file.OldPath, file.Path)
    }
    if file.IsNew {
        fmt.Fprintf(&buf, "--- /dev/null\n")
    } else {
        fmt.Fprintf(&buf, "--- a/%s\n", aPath) // use aPath (OldPath for renames)
    }
    if file.IsDeleted {
        fmt.Fprintf(&buf, "+++ /dev/null\n")
    } else {
        fmt.Fprintf(&buf, "+++ b/%s\n", file.Path)
    }

    var delta int64 = 0

    for i, hunk := range allHunks {
        if !selected[i] {
            // Skipped hunk: accumulate its effect on line numbers
            delta += hunk.OldCount - hunk.NewCount
        } else {
            // Selected hunk: emit with adjusted new_start
            adjustedNewStart := hunk.NewStart + delta

            fmt.Fprintf(&buf, "@@ -%d,%d +%d,%d @@\n",
                hunk.OldStart, hunk.OldCount,
                adjustedNewStart, hunk.NewCount)

            for _, line := range hunk.Lines {
                switch line.Op {
                case OpDelete:
                    fmt.Fprintf(&buf, "-%s\n", line.Content)
                case OpAdd:
                    fmt.Fprintf(&buf, "+%s\n", line.Content)
                }
            }
        }
    }

    return buf.Bytes(), nil
}
```

### 8.5 Content Fingerprinting (Go)

```go
// diff/fingerprint.go
// Lines are in original order (NOT sorted) to avoid false collisions
// from reordered code. See Section 3.8 for the canonical algorithm.
func Fingerprint(hunk Hunk) string {
    h := sha256.New()
    for _, line := range hunk.Lines {
        if line.Op == OpDelete {
            h.Write([]byte("-" + line.Content + "\n"))
        }
    }
    h.Write([]byte("||"))
    for _, line := range hunk.Lines {
        if line.Op == OpAdd {
            h.Write([]byte("+" + line.Content + "\n"))
        }
    }
    return hex.EncodeToString(h.Sum(nil))
}
```

### 8.6 Execution Flow (Go)

```go
// cli/run.go
func Run(planJSON []byte) (*Result, error) {
    // === Phase 1: Validate Everything ===

    // 1. Environment checks
    if err := git.EnsureRepo(); err != nil {
        return nil, ValidationError(err) // exit 2
    }
    if err := git.EnsureCleanStaging(); err != nil {
        return nil, ValidationError(err) // exit 2
    }

    // 2. Parse plan
    p, err := plan.Parse(planJSON)
    if err != nil {
        return nil, ValidationError(err)
    }

    // 3. Intent-to-add for untracked files with hunks
    addedN, err := intentToAddNewFiles(p)
    if err != nil {
        return nil, ValidationError(err)
    }
    // Phase 1 validation wrapper: captures diff, validates plan, runs
    // sequential dry-run. Reverts git add -N on ANY Phase 1 error.
    // Phase 2 needs the intent-to-add files, so do NOT revert on success.
    origFiles, err := runPhase1Validation(p, addedN)
    if err != nil {
        revertIntentToAdd(addedN)
        return nil, err
    }
    // runPhase1Validation internally:
    //   4. git.Diff("-U0", "-M") -> parse -> index original hunks
    //   5. plan.Validate(p, origFiles) -- coverage, duplicates, ranges
    //   6. dryRunWithTempIndex(p, origFiles) -- sequential apply to temp index

    // === Phase 2: Execute Deterministically ===
    // Intent-to-add files remain visible for Phase 2.
    // After successful commits, the intent-to-add state is consumed.

    result := &Result{
        Total:   len(p.Commits),
        Commits: make([]CommitResult, 0, len(p.Commits)),
    }

    for i, commit := range p.Commits {
        cr, err := executeCommit(i, commit, origFiles)
        if err != nil {
            // Differentiate: only reset staging on apply failure, not commit failure
            if isApplyError(err) {
                git.Run("reset", "HEAD", "--") // unstage partial staging
            }
            // On commit failure (e.g., hook): leave staging intact
            revertRemainingIntentToAdd(addedN, result.Commits) // cleanup orphaned -N entries
            cr.Status = "failed"
            cr.Error = err.Error()
            result.Commits = append(result.Commits, cr)
            result.Error = err.Error()
            if i == 0 {
                result.Hint = "No commits were created. Fix the issue and retry."
            } else {
                result.Hint = fmt.Sprintf(
                    "Fix the issue. Commits 0-%d are done. Re-plan remaining changes.", i-1)
            }
            return result, err
        }
        cr.Status = "committed"
        result.Committed++
        result.Commits = append(result.Commits, cr)
    }

    return result, nil
}
```

## 9. Skill Design

### 9.1 Skill Metadata

```yaml
---
name: hc
description: Hunk-based atomic git commits for AI agents. Splits large diffs into precise, atomic commits by selecting specific diff hunks per commit.
---
```

### 9.2 Activation

This skill activates when:
- The agent needs to create atomic commits from uncommitted changes
- The user asks for hunk-level commit granularity
- The agent has written multiple logical changes (e.g., several tests, feature + test, refactor + fix)

### 9.3 Workflow

```
# Step 1: See what changed (indexed hunks)
ac diff --json

# Step 2: Write a commit plan
# Map hunks to commits based on logical grouping.
# Use the ORIGINAL diff's hunk indices -- ac handles re-indexing.
# IMPORTANT: Every hunk must be assigned to exactly one commit.

# Step 3: Execute
echo '<plan-json>' | ac run -

# Or with a file:
ac run plan.json

# Optional: validate first
echo '<plan-json>' | ac run --dry-run -
```

### 9.4 Plan Writing Rules

1. **Run `ac diff --json` once.** Use the indexed output to see all files and hunk indices.
2. **Assign EVERY hunk to exactly one commit.** Complete coverage is required. No hunk can be left unassigned.
3. **Use original indices.** Always reference hunks by their position in the `ac diff` output, even for later commits.
4. **Full-file for simple cases.** If an entire file belongs in one commit, omit `hunks`.
5. **Group by logical change.** One test per commit, one feature per commit, etc.
6. **Conventional commit messages.** Use the project's commit convention.
7. **Use `allow_unplanned` sparingly.** Only for files with WIP changes that should not be committed yet.

### 9.5 Common Patterns

**One test per commit (5 tests in one file):**
```json
{
  "commits": [
    {"message": "test(auth): add token expiry test", "files": [{"path": "auth_test.go", "hunks": [0]}]},
    {"message": "test(auth): add token refresh test", "files": [{"path": "auth_test.go", "hunks": [1]}]},
    {"message": "test(auth): add token revoke test", "files": [{"path": "auth_test.go", "hunks": [2]}]},
    {"message": "test(auth): add token rotate test", "files": [{"path": "auth_test.go", "hunks": [3]}]},
    {"message": "test(auth): add token validate test", "files": [{"path": "auth_test.go", "hunks": [4]}]}
  ]
}
```

**Feature + test across files:**
```json
{
  "commits": [
    {
      "message": "feat(auth): add refresh endpoint",
      "files": [
        {"path": "auth.go", "hunks": [0, 1]},
        {"path": "handler.go"}
      ]
    },
    {
      "message": "test(auth): add refresh endpoint tests",
      "files": [
        {"path": "auth_test.go"},
        {"path": "handler_test.go"}
      ]
    }
  ]
}
```

**Partial commit with WIP excluded:**
```json
{
  "allow_unplanned": ["experiments/new_idea.go"],

    {
      "message": "fix(db): close connections on timeout",

        {"path": "db.go", "hunks": [0]},

    }
  ]

```

### 9.6 Error Recovery

ac is designed so errors only happen during validation (before any commits):
1. Read the error and hint from ac's JSON output
2. Fix the plan (adjust hunk indices, add missing files, etc.)
3. Retry: `echo '<fixed-plan>' | ac run -`

No commits have been created, no git state has changed. Simple retry.

## 10. Test Strategy

### 10.1 Unit Tests

**Diff Parsing (`diff/parse_test.go`):**
1. Parse `@@ -10,3 +10,5 @@` header -- correct old/new start/count
2. Parse `@@ -5 +10 @@` header -- omitted count defaults to 1
3. Parse `@@ -15,0 +16,2 @@` header -- pure addition (0 old lines)
4. Parse `@@ -15,3 +15,0 @@` header -- pure deletion (0 new lines)
5. Parse multi-file diff -- correct file boundaries
6. Parse new file diff (`--- /dev/null`)
7. Parse deleted file diff (`+++ /dev/null`)
8. Parse binary file diff -- `IsBinary` flag set
9. Parse rename diff -- old/new paths captured
10. Parse mode change diff -- `ModeChange` captured
11. Handle `\ No newline at end of file` marker -- not counted in line counts
12. Handle empty diff -- returns empty slice, no error

**Content Fingerprinting (`diff/fingerprint_test.go`):**
13. Same content, different line numbers -- same fingerprint
14. Different content -- different fingerprint
15. Same additions, different deletions -- different fingerprint
16. Order-dependent -- shuffled lines produce different fingerprint
17. Empty hunk -- stable fingerprint (empty string hash)

**Hunk Matching (`diff/match_test.go`):**
18. All hunks present -- 1:1 mapping
19. Some hunks committed (fewer hunks in current diff) -- remaining matched by fingerprint
20. Identical content hunks -- disambiguated by old_start proximity
21. No matching hunk found -- error with hint

**Delta Accumulation (`patch/build_test.go`):**
22. Select all hunks -- no adjustment needed, delta stays 0
23. Select first hunk only -- no adjustment (no prior skipped hunks)
24. Select last hunk only -- delta accumulated from all skipped hunks
25. Select non-contiguous [0, 2] from 4 hunks -- hunk 2's new_start adjusted by skipped hunk 1's delta
26. Select non-contiguous [1, 3] from 4 hunks -- both adjusted correctly
27. Skip hunk that adds lines (new_count > old_count) -- delta negative (subsequent hunks shift up)
28. Skip hunk that deletes lines (old_count > new_count) -- delta positive (subsequent hunks shift down)
29. Skip hunk with equal old/new count -- delta unchanged

### 10.3 Test Methodology

All integration tests follow this pattern:

```go
func TestScenario(t *testing.T) {
    // 1. Create temp directory with git init
    dir := t.TempDir()
    git(t, dir, "init")
    git(t, dir, "commit", "--allow-empty", "-m", "initial")

    // 2. Create base files and commit
    writeFile(t, dir, "auth.go", baseContent)
    git(t, dir, "add", "auth.go")
    git(t, dir, "commit", "-m", "base")

    // 3. Make changes (the "working tree" state)
    writeFile(t, dir, "auth.go", modifiedContent)

    // 4. Run hc with plan
    result := runAC(t, dir, plan)

    // 5. Assert commits exist with correct content
    assertCommitCount(t, dir, expectedCount)
    assertCommitMessage(t, dir, 0, "expected message")
    assertCommitDiff(t, dir, 0, expectedDiff)
}
```

## 12. What This Spec Does NOT Cover

- **Amending commits.** hc creates new commits only.
- **Interactive mode.** No prompts, no TUI. Agents cannot drive interactive UIs.
- **Commit signing.** If the user has `commit.gpgSign` configured, git handles it. hc does not add/remove signing.
- **Pre-commit hooks.** hc calls `git commit` normally. Hooks run. If a hook fails, the commit fails and hc reports it. hc never uses `--no-verify`.
- **Remote operations.** No push, no pull, no fetch.
- **Multiple plan files / directory input.** Deferred. One plan per invocation is sufficient for v0.1.
- **Rebase / squash.** Out of scope. hc creates commits; the agent can rebase afterward if needed.
- **Merge conflict resolution.** hc expects a clean working tree (no conflicts).
